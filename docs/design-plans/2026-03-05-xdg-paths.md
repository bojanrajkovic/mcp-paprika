# XDG Paths Design

## Summary

This module creates a single, authoritative source for platform-native filesystem paths within the `mcp-paprika` server. Rather than hard-coding paths or reimplementing the logic to read XDG environment variables, it delegates to the `env-paths` npm package, which handles Linux (XDG Base Directory Specification), macOS (Library folders), and Windows (AppData) conventions automatically. The result is a thin wrapper bound to the application name `mcp-paprika` that exposes five functions — one per directory category — each returning an absolute path string computed at module load time.

The design is intentionally minimal: no I/O, no async, no branching logic. `env-paths` is called once when the module is first imported, and the resolved paths are held as a frozen object. Consumers such as the configuration loader and the disk cache call the relevant function to obtain a path, then handle any directory creation themselves. This separation keeps the path-resolution concern isolated in a single leaf module with no internal dependencies.

## Definition of Done

`src/utils/xdg.ts` wraps `env-paths` v4 with app name `mcp-paprika` (no suffix), providing platform-native application directory paths. Exports 5 functions (`getConfigDir()`, `getCacheDir()`, `getDataDir()`, `getLogDir()`, `getTempDir()`) that return absolute `string` paths synchronously with no I/O. `env-paths` is added as a runtime dependency. Tests verify paths end with `/mcp-paprika` (or platform equivalent) and are absolute. On Linux, tests verify XDG env vars are honored. Directory creation is out of scope (responsibility of P1-U04 and P1-U08).

## Glossary

- **XDG Base Directory Specification**: A freedesktop.org standard that defines environment variables (`$XDG_CONFIG_HOME`, `$XDG_CACHE_HOME`, etc.) governing where applications should store config, cache, and data files on Linux. When the variables are unset or empty, the spec prescribes fallback paths under `~/.config`, `~/.cache`, and so on.
- **env-paths**: A small npm package that resolves platform-native application directory paths — XDG paths on Linux, `~/Library/…` on macOS, and `%APPDATA%/…` on Windows — given an application name.
- **suffix**: An `env-paths` option that, by default, appends `-nodejs` to the application name. Passing `{ suffix: '' }` disables this so the path segment is exactly `mcp-paprika`.
- **leaf dependency**: A module in `src/utils/` that is prohibited from importing anything from other `src/` modules. It may only depend on npm packages or Node.js built-ins, ensuring no risk of circular dependencies.
- **frozen object**: A JavaScript object on which `Object.freeze()` has been called, making its properties immutable at runtime. Used here to describe the path object returned by `env-paths`.
- **P1-U04 / P1-U08**: Internal unit identifiers in the project's implementation plan. P1-U04 is the configuration loader; P1-U08 is the disk cache. Both are downstream consumers of this module.
- **`||` vs `??` (nullish coalescing)**: Two ways to provide a fallback value. `??` treats only `null` and `undefined` as "missing", while `||` also treats an empty string `""` as "missing". The XDG spec says an empty `$XDG_CONFIG_HOME` should be treated as unset, so `||` semantics are correct here.

## Architecture

Thin wrapper module around `env-paths` v4 that binds the app name `mcp-paprika` and re-exports platform-native directory paths as named functions.

`env-paths` resolves paths per platform convention:

| Platform | Config                                                    | Cache                                                   |
| -------- | --------------------------------------------------------- | ------------------------------------------------------- |
| Linux    | `$XDG_CONFIG_HOME/mcp-paprika` or `~/.config/mcp-paprika` | `$XDG_CACHE_HOME/mcp-paprika` or `~/.cache/mcp-paprika` |
| macOS    | `~/Library/Preferences/mcp-paprika`                       | `~/Library/Caches/mcp-paprika`                          |
| Windows  | `%APPDATA%\mcp-paprika\Config`                            | `%LOCALAPPDATA%\mcp-paprika\Cache`                      |

The module calls `envPaths('mcp-paprika', { suffix: '' })` once at module load time. The result is a frozen object of strings — no I/O, no async. Each exported function returns the corresponding property.

**Contract:**

```typescript
export function getConfigDir(): string;
export function getCacheDir(): string;
export function getDataDir(): string;
export function getLogDir(): string;
export function getTempDir(): string;
```

All functions are synchronous, return absolute paths, and perform no filesystem operations. Callers are responsible for creating directories as needed.

**Downstream consumers:**

- P1-U04 (configuration loader) — calls `getConfigDir()` to locate `config.json`
- P1-U08 (disk cache) — calls `getCacheDir()` to locate recipe JSON files and `index.json`

## Additional Considerations

**Deviation from unit spec:** The original P1-U03 spec prescribed a custom implementation using `node:os` and `node:path` with `??` (nullish coalescing). This design replaces that with `env-paths` v4 for three reasons: (1) cross-platform support (Windows, macOS native paths), (2) `||` semantics matching the XDG spec's treatment of empty env vars as "not set", (3) maintained library handling platform edge cases.

**No dedicated tests:** The module contains no branching logic — it's a configuration binding that calls `env-paths` with a string literal and re-exports properties. Downstream modules (P1-U04, P1-U08) provide integration coverage. Tests can be added later if the module grows more complex.

**App name consistency:** The string `'mcp-paprika'` appears in one place (the `envPaths()` call). If the app name ever changes, only this file needs updating.
