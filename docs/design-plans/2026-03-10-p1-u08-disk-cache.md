# Disk Cache Design

## Summary

`DiskCache` is a new persistence layer that stores the Paprika recipe library on the local filesystem between MCP server restarts. Without it the server must fetch every recipe and category from the Paprika Cloud Sync API on every startup, which is slow and hammers a rate-limited remote service. With it the server loads previously fetched data from disk, calls the API only for entries whose content hash has changed, and writes results back atomically before the session ends.

The implementation lives in `src/cache/disk-cache.ts` as a single class alongside the existing in-memory `RecipeStore`. It maintains an `index.json` file that maps each recipe and category UID to its content hash — a compact summary used to detect changes without reading every JSON file. Individual recipe and category objects are stored as separate JSON files under `recipes/` and `categories/` subdirectories. Writes are fully deferred: `put*()` calls buffer data in memory and `flush()` writes everything in a single fsynced batch, then commits the index via an atomic rename. A companion schema change to `src/paprika/types.ts` splits each existing Zod schema into an API-facing variant (validates snake_case wire format, transforms to camelCase) and a storage-facing variant (validates the camelCase JSON already on disk), ensuring the same Zod-enforced types are used at both boundaries.

## Definition of Done

## Glossary

- **UID**: Unique identifier string for a recipe or category, as assigned by the Paprika Cloud Sync API. Represented in the codebase as a branded string type (`RecipeUid`, `CategoryUid`) so the compiler prevents mixing the two kinds.
- **hash**: An opaque string provided by the Paprika API alongside each recipe and category entry. When a recipe is edited in Paprika its hash changes. `DiskCache` stores hashes in `index.json` and compares them against the remote listing to determine which entries are new, changed, or deleted — without reading full recipe bodies for every entry.
- **`CacheIndex`**: The in-memory representation of `index.json`. A plain object with two `uid → hash` maps, one for recipes and one for categories. File-local to `disk-cache.ts` and not exported.
- **`RecipeEntry` / `CategoryEntry`**: Lightweight objects returned by the Paprika list endpoints containing only a `uid` and a `hash`. Used as the input to `diffRecipes()` / `diffCategories()` to avoid fetching full recipe bodies before knowing what has changed.
- **`DiffResult`**: The output of a diff method — three arrays of UIDs (`added`, `changed`, `removed`) classifying how the remote listing differs from the local index.
- **`RecipeStoredSchema` / `CategoryStoredSchema`**: New Zod schemas introduced in Phase 1 that validate camelCase JSON as read back from disk. Distinct from `RecipeSchema` / `CategorySchema`, which accept snake_case wire format and transform it.
- **`recipeCamelShape`**: A plain Zod shape object (the argument to `z.object()`) that defines all recipe fields in camelCase. Shared between `RecipeStoredSchema` (disk validation) and the transform output of `RecipeSchema` (API), making them the same TypeScript type by construction.
- **atomic rename / temp-then-rename**: A write strategy where data is written to a temporary file first, then `rename()` moves it over the target. On POSIX filesystems a rename within the same directory is atomic — readers see either the old file or the new one, never a partial write.
- **fsync**: A system call that flushes a file's data from the OS kernel's write-back cache to durable storage. Used here via Node.js `FileHandle.sync()` so that recipe and category files are guaranteed to be on disk before the index rename commits them.
- **pending map (`_pendingRecipes`, `_pendingCategories`)**: In-memory `Map` instances that hold writes buffered by `put*()` calls but not yet written to disk. Drained by `flush()`. `get*()` checks these maps before falling back to disk so a caller can read back data it just put in the same batch.
- **imperative shell**: An architectural pattern (functional core, imperative shell) where all I/O and side effects are concentrated in one outer layer while business logic stays pure. `DiskCache` is explicitly this — it owns all file I/O for the cache and contains no business logic.
- **Zod**: A TypeScript-first schema validation library used throughout the project to validate external data at system boundaries. Schemas can include `.transform()` steps to reshape data (e.g. snake_case → camelCase) at parse time.
- **Zod `.safeParse()`**: The non-throwing variant of Zod validation. Returns `{ success: true, data }` or `{ success: false, error }` instead of throwing on invalid input. Used here so a corrupt `index.json` can be detected and recovered from without an uncaught exception.
- **XDG Base Directory Specification**: A freedesktop.org standard defining where applications store config, cache, and data on Linux. `$XDG_CACHE_HOME` (typically `~/.cache`) is where `DiskCache` places its directory, resolved via `src/utils/xdg.ts`.
- **`FileHandle`**: The Node.js object returned by `fs/promises.open()`. Exposes `writeFile()`, `sync()` (fsync), and `close()` as methods on an already-open file descriptor, enabling the write → fsync → close sequence without reopening the file.
- **`mkdtemp` / `rm`**: Node.js `fs/promises` functions used in tests to create a fresh temporary directory before each test case and remove it afterwards, giving each test an isolated real filesystem.

## Architecture

`DiskCache` is the imperative shell for all cache persistence in `src/cache/disk-cache.ts`. It stores full recipe and category JSON on disk and maintains an in-memory `CacheIndex` (a `uid → hash` map for each entity type) that drives efficient sync diffing. No business logic lives here — only I/O coordination.

### On-disk layout

```
$XDG_CACHE_HOME/paprika-mcp/
├── index.json               ← CacheIndex (uid → hash for recipes + categories)
├── recipes/
│   └── {uid}.json           ← Full Recipe JSON (camelCase, pretty-printed)
└── categories/
    └── {uid}.json           ← Full Category JSON (camelCase, pretty-printed)
```

### Schema changes to `src/paprika/types.ts`

`RecipeSchema` and `CategorySchema` have snake_case → camelCase transforms, making them unsuitable for validating the camelCase JSON stored on disk. The solution is to define the camelCase output shape once, then derive both the stored schema and the API schema from it:

```typescript
// Single source of truth for camelCase field definitions
const recipeCamelShape = { uid: RecipeUidSchema, hash: z.string(), name: z.string(), /* ... all 28 camelCase fields */ };

// Stored schema — validates camelCase JSON from disk, no transform
export const RecipeStoredSchema = z.object(recipeCamelShape);

// Recipe type derived from stored schema (same shape as before)
export type Recipe = z.infer<typeof RecipeStoredSchema>;

// API schema — validates snake_case from wire, transforms into Recipe
export const RecipeSchema = z.object({ /* snake_case fields */ }).transform((...): Recipe => ({ /* renames */ }));
```

The `: Recipe` annotation on the transform return ties `RecipeSchema`'s output to `RecipeStoredSchema`'s shape at compile time. `CategoryStoredSchema` follows the same pattern (only `orderFlag` and `parentUid` are renamed).

### `DiskCache` class contract

Public API is unchanged from the original design. Internally, `putRecipe()` / `putCategory()` now buffer into two private pending maps (`_pendingRecipes`, `_pendingCategories`) rather than writing to disk immediately. `flush()` drains those maps.

```typescript
export class DiskCache {
  constructor(cacheDir: string, log?: (msg: string) => void);
  async init(): Promise<void>;
  async flush(): Promise<void>;
  async getRecipe(uid: string): Promise<Recipe | null>;
  async putRecipe(recipe: Recipe, hash: string): Promise<void>;
  async removeRecipe(uid: string): Promise<void>;
  async getAllRecipes(): Promise<Recipe[]>;
  async getCategory(uid: string): Promise<Category | null>;
  async putCategory(category: Category, hash: string): Promise<void>;
  diffRecipes(entries: RecipeEntry[]): DiffResult; // sync
  diffCategories(entries: CategoryEntry[]): DiffResult; // sync
}
```

`log` defaults to a no-op. In Phase 2, the entry point passes the MCP server's `sendLoggingMessage` function.

### Key algorithms

**`init()`:** Creates `recipes/` and `categories/` subdirectories (`mkdir -p`), then loads `index.json` using `CacheIndexSchema.safeParse()`. ENOENT → empty index (first run). Parse failure → `log()` warning + empty index (corruption recovery). Other I/O error → rethrow.

**`flush()` — fully-deferred, fsynced batch write:**

All pending recipe and category files are written in parallel using Node.js `FileHandle` (`open → writeFile → sync → close`), ensuring each file is fsynced before the index is committed. Then the index is written atomically with the same pattern before being renamed into place:

```
Promise.all(
  pendingRecipes: open(recipes/{uid}.json, 'w') → fh.writeFile → fh.sync → fh.close
  pendingCategories: open(categories/{uid}.json, 'w') → fh.writeFile → fh.sync → fh.close
)
open(cacheDir/.index-{ts}.tmp, 'w') → fh.writeFile → fh.sync → fh.close
rename(.index-{ts}.tmp → index.json)   ← atomic commit
pendingRecipes.clear(), pendingCategories.clear()
```

Fsyncing each recipe/category file before the index rename ensures that if a crash occurs between writes, the index will never reference files that are not durably on disk.

**`_diffEntries(remote, local)`:** Private synchronous method. Builds a `Set` of remote UIDs for O(1) lookup, then classifies each remote entry as `added` or `changed` and each local-only UID as `removed`. Both `diffRecipes()` and `diffCategories()` assert `_index !== null` and delegate here.

**`putRecipe()` / `putCategory()`:** Add to `_pendingRecipes` / `_pendingCategories` map and update `_index` in memory. No file I/O. The caller (sync engine) calls `flush()` once per batch.

**`getRecipe()` / `getCategory()`:** Check pending map first (returns buffered value if present); otherwise read file from disk, ENOENT → `null`, otherwise `JSON.parse` → `RecipeStoredSchema.parse()` / `CategoryStoredSchema.parse()`.

**`getAllRecipes()`:** Returns all recipes from pending map merged with disk reads from `recipesDir` (pending entries shadow on-disk entries for the same UID). ENOENT on the directory → `[]`. Each disk file is validated with `RecipeStoredSchema.parse()`.

### Internal type

`CacheIndexSchema` and `CacheIndex` are file-local to `disk-cache.ts` and not exported:

```typescript
const CacheIndexSchema = z.object({
  recipes: z.record(z.string(), z.string()),
  categories: z.record(z.string(), z.string()),
});
type CacheIndex = z.infer<typeof CacheIndexSchema>;
```

## Existing Patterns

**ENOENT detection:** `config.ts` defines a local `isNodeError(error): error is NodeJS.ErrnoException` guard (`error instanceof Error && 'code' in error`). `disk-cache.ts` defines the same helper locally — it is not exported from utils.

**Real filesystem in tests:** `config.test.ts` uses `mkdtempSync` / `rmSync` for a real temp directory. `disk-cache.test.ts` uses the async equivalents (`mkdtemp` / `rm`) since all DiskCache operations are async. No mock filesystem.

**Schema-first validation:** All external/untrusted data is validated with Zod schemas before use. `disk-cache.ts` extends this to disk reads via `RecipeStoredSchema` and `CategoryStoredSchema`.

**New pattern introduced — camelCase output schema:** Splitting the Zod schema into `recipeCamelShape` (output definition) + `RecipeStoredSchema` (disk validation) + `RecipeSchema` (API transform) is new to this codebase. The `: Recipe` annotation on `RecipeSchema`'s transform return enforces consistency at compile time.

## Additional Considerations

**Fully-deferred writes** — `putRecipe()` and `putCategory()` perform no file I/O. All writes happen in `flush()`, which also fsyncs every file before committing the index. This means `recipes/{uid}.json` and `categories/{uid}.json` do not exist until `flush()` is called.

**`getRecipe()` checks pending map first** — since recipe files don't exist until flush, reads must consult the pending map before falling back to disk. This preserves correct behaviour if a caller reads back a recipe it just put in the same sync cycle (unlikely in practice, but correct by design).

**`getAllRecipes()` merges pending and disk** — pending entries shadow any on-disk entries for the same UID. This is important at startup: if a prior session wrote files but crashed before flushing the index, `getAllRecipes()` still returns those files correctly.

**No `removeCategory()` or `getAllCategories()`** — categories are synced wholesale each cycle (`setCategories()` on RecipeStore). The cache only needs per-category get/put for building the stored index; bulk removal is handled by diffCategories detecting removed UIDs, which the sync engine then skips caching for.

**Temp file atomicity is guaranteed** — the temp file is written to `cacheDir` (not `os.tmpdir()`), ensuring it is on the same filesystem as `index.json`. POSIX `rename()` across the same filesystem is atomic.
