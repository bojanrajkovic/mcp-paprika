# Test Requirements: project-conventions

## Overview

Maps all 17 acceptance criteria from the project-conventions design unit to verification methods. This design unit is infrastructure (dev tooling and documentation) with no application code. Acceptance criteria are verified operationally — commands succeed or fail, configuration files contain expected content — rather than by unit tests.

Verification falls into two categories:

- **Automated:** Shell commands that produce a deterministic pass/fail exit code.
- **Human:** Content review of documentation files where correctness requires judgment.

| project-conventions.AC4.1 | `.oxlintrc.json` configures `eqeqeq` as error                                                       | Inspect `.oxlintrc.json` rules object for `"eqeqeq": "error"`                                                              | `eqeqeq` rule is set to `"error"`                                 |

| project-conventions.AC4.3 | `.oxlintrc.json` does NOT configure `no-unused-vars` (handled by tsc via `@tsconfig/strictest`)     | Inspect `.oxlintrc.json` rules object for absence of `no-unused-vars`                                                      | `no-unused-vars` key is absent from rules                         |

### Automated Verification Notes

- **AC2.2 and AC2.5** require a git staging area. Verification scripts must create temporary files, stage them, run the hook, inspect results, then clean up with `git reset` and `rm`.
- **AC3.5** requires temporarily corrupting a file, running the check, then restoring with `git checkout`.
- **AC4.1–AC4.3** are configuration file assertions verified by inspecting JSON content.

## Human Verification

| Criterion                 | Text                                                                                                                                                                                                      | Justification                                                                                                                                                                                                    | Verification Approach                                                                                                                                                                                                                                   |
| ------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| project-conventions.AC1.1 | Root CLAUDE.md exists and covers all convention categories: tech stack, commands, code conventions (imports, TypeScript style, error handling), dependency policy, testing, git conventions, version sync | Content completeness and accuracy is a qualitative judgment — a command can check for heading presence but not whether content under each heading is correct and sufficient                                      | Read `CLAUDE.md` and confirm all categories are present as sections with substantive content: Tech Stack, Commands, Code Conventions (Imports and Modules, TypeScript Style, Error Handling), Dependency Policy, Testing, Git Conventions, Version Sync |
| project-conventions.AC1.2 | All 7 package.json scripts are documented in the Commands section                                                                                                                                         | The design says "7 scripts" but implementation adds 10 (7 original + typecheck, test:watch, lint:fix from guidance). Verifying completeness requires cross-referencing the Commands table against `package.json` | Read the Commands section in `CLAUDE.md`, read `package.json` scripts, and verify every script in `package.json` appears in the Commands table with an accurate description                                                                             |
| project-conventions.AC1.3 | Dependency policy states "minimize runtime dependencies" (not "zero dependencies")                                                                                                                        | The exact wording matters — this is a content correctness check on a specific phrase                                                                                                                             | Read the Dependency Policy section and confirm it uses the phrase "Minimize runtime dependencies" or equivalent (not "zero dependencies")                                                                                                               |
| project-conventions.AC1.4 | Domain-level CLAUDE.md stubs exist in all 7 `src/` subdirectories with Purpose, Contracts, and Dependencies sections                                                                                      | File existence can be automated, but verifying each stub has meaningful Purpose text and correct Dependencies boundaries requires judgment                                                                       | Confirm files exist at `src/{paprika,cache,tools,resources,features,types,utils}/CLAUDE.md`. Open each and verify it contains Purpose, Contracts, and Dependencies sections with content appropriate to the module                                      |

### Human Verification Notes

- **AC1.1** could be partially automated by checking for heading strings, but this would not verify that content under each heading is accurate, complete, or useful.
- **AC1.2** has a nuance: the design says "7" but implementation adds 10 scripts. The reviewer should verify the Commands table documents all scripts that actually exist in `package.json`.
- **AC1.4** file existence could be scripted, but verifying the sections have appropriate content for each module requires human review.
