# Paprika Type Definitions ‚Äî Phase 1: Branded UIDs, Entry Schemas, and Dependencies

**Goal:** Install type-fest, create `src/paprika/types.ts` with branded UID schemas and entry schemas for the sync list endpoints.

**Architecture:** Schema-first design using Zod as the single source of truth. Branded types via `z.brand()` create compile-time distinct UID types with zero runtime overhead. Entry schemas model the Paprika API sync list response shape (`{uid, hash}`).

**Tech Stack:** TypeScript 5.9, Zod 3.25.x, type-fest (dev dependency)

**Scope:** 3 phases from original design (phase 1 of 3)

**Codebase verified:** 2026-03-03

---

## Acceptance Criteria Coverage

This phase implements and tests:

### paprika-types.AC1: Zod schemas validate API responses

- **paprika-types.AC1.1 Success:** RecipeEntrySchema parses `{uid: "abc", hash: "def"}` and returns branded RecipeUid
- **paprika-types.AC1.8 Failure:** RecipeEntrySchema rejects response with non-string uid

- `/home/brajkovic/Projects/mcp-paprika/src/paprika/CLAUDE.md` ‚Äî module boundary contracts

**Implementation:**

Create `src/paprika/types.ts` as a leaf dependency importing only from `zod`. This file is part of the Functional Core ‚Äî pure type definitions and schemas with no I/O or side effects.

The file must contain:

1. **Branded UID schemas** using `z.string().brand()`:
   - `RecipeUidSchema` ‚Äî branded with `"RecipeUid"`
   - `CategoryUidSchema` ‚Äî branded with `"CategoryUid"`

2. **Derived UID types** via `z.infer<>`:
   - `RecipeUid` ‚Äî `string & { __brand: "RecipeUid" }`
   - `CategoryUid` ‚Äî `string & { __brand: "CategoryUid" }`

3. **Entry schemas** for sync list endpoints (both have `uid` + `hash`):
   - `RecipeEntrySchema` ‚Äî `z.object({ uid: RecipeUidSchema, hash: z.string() })`
   - `CategoryEntrySchema` ‚Äî `z.object({ uid: CategoryUidSchema, hash: z.string() })`

4. **Derived entry types** via `z.infer<>`:
   - `RecipeEntry`
   - `CategoryEntry`

All schemas and types must be named exports. No default exports.
